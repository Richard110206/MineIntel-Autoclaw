# 虚拟多专家编排流程

该文件用于让 AutoClaw/GLM 在一个 Skill 内模拟原 MineIntel 的多智能体流程。不要把这些角色实现成固定脚本，角色判断、取舍和综合由模型完成。

## 角色分工

1. 矿井应用专家：负责查询 `mineintel-application-kg` 知识图谱，把用户输入映射到矿井场景、痛点、解决方案、技术设备和白皮书/蓝皮书依据。
2. 领域分析师：优先调用 MCP tool `mineintel_domain_analyst_search`，检索煤炭学报、采矿与安全工程学报、工矿自动化、煤炭科学技术等中文矿业应用论文线索。
3. 行业前沿技术专家：优先调用 MCP tool `mineintel_frontier_technology_search`，检索 arXiv、DBLP、近年前沿论文/综述，并返回一个 GitHub baseline。
4. 导师与申报专家：负责把技术方向匹配到导师方向样例，并形成大创/科研申报可落地的题目、技术路线和风险控制。
5. 科研经验参考专家：负责检索知乎/小红书公开经验线索，提炼项目推进、答辩和风险控制建议，不作为论文或技术事实依据。
6. 总报告专家：负责把前面各专家结果合并为报告正文，再调用报告交付父级 Skill 生成 HTML 完整报告、逐页展示 Deck、文献综述 LaTeX 和可选 PDF。

## 推荐执行顺序

先由矿井应用专家调用 `kg_search.py` 形成问题边界。再由领域分析师调用 MCP 的 `mineintel_domain_analyst_search` 检索中文应用论文。随后行业前沿技术专家调用 MCP 的 `mineintel_frontier_technology_search` 检索国际前沿和一个 GitHub baseline。再由导师与申报专家调用导师检索脚本匹配导师和项目建议。最后补充科研经验参考，并由总报告专家调用 `save_report.py` 生成交付文件。

## 信息合并规则

- 知识图谱用于稳定补全矿业场景、痛点、解决方案、技术设备和来源依据。
- 公开搜索结果只作为线索，题名、作者、年份、期刊名不能凭空补全。
- GitHub baseline 只保留一个最相关仓库，重点看任务相关性、最近更新、star、训练脚本和自定义数据支持。
- 导师推荐来自候选库时只展示姓名、学院和链接，不额外加来源标签。
- 最终报告要把“明确结果”和“检索线索”分开写。

## 与原版 MineIntel 的对应

原版多智能体是代码层面的 Agent 编排；本 Skill 版把多 Agent 角色压缩到 `SKILL.md` 和本参考文件中，由 AutoClaw/GLM 在执行时完成角色切换、工具选择和综合判断。
